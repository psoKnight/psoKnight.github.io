> eino「逐能力核对」系列第 1 篇。系列按「掌握 → 学习 → 企业级」三阶段,每项能力单独一篇,只看 **v0.8.12** 真实源码给结论——实现了就是 ✅,没实现就是 ❌,含糊的一律 ⚠️ 划清边界。作为开篇,本文先用一节把三层架构与「流优先」讲透(后续 11 篇都会引用),再从一次线上事故切入 Prompt 这一环。结论:**✅ 一等实现,但它真正的价值不在「模板」,在于确立了一条贯穿全框架的类型主线。**

## 先建立坐标:三层架构 + 流优先(全系列公共背景)

eino 是字节跳动 CloudWeGo 开源的 Go 语言 LLM 应用框架。工程上它最克制的一点,是把整个框架切成三层,每层只依赖下一层的**接口**,不依赖实现:

```
┌─────────────────────────────────────┐
│  adk (Agent Development Kit)         │  自主决策:ReAct / Supervisor / Skill ...
├─────────────────────────────────────┤
│  compose (Graph / Chain 编排)        │  确定性流程:类型对齐 + 流缝合
├─────────────────────────────────────┤
│  components (ChatModel/Tool/Prompt…) │  组件抽象:可插拔的接口
└─────────────────────────────────────┘
         schema (Message / StreamReader)   贯穿所有层的数据类型
```

贯穿三层的是 `schema` 里两个类型。第一个是 `Message`,**框架里所有「一段对话」的唯一表示**:

```go
type Message struct {
	Role       schema.RoleType  // system / user / assistant / tool
	Content    string
	ToolCalls  []schema.ToolCall
	ToolCallID string
	// ...
}
```

第二个是泛型流 `StreamReader[T]`——eino 敢称「流优先」的载体。它不是某个组件的附属功能,而是和同步调用**并列的一等公民**:

```go
func (sr *StreamReader[T]) Recv() (T, error)  // io.EOF 表示流结束
func (sr *StreamReader[T]) Close()
```

记住这两个类型,因为全系列的核心论点之一就是:**eino 的每个组件,输入输出都收敛到 `Message` / `StreamReader[Message]` 上**。这种收敛不是洁癖,它是后面 compose 层能自动缝合流与非流([第 5 篇]({% post_url 2026-07-09-eino-05-compose %}))、ReAct 能把图封装成 Agent([第 6 篇]({% post_url 2026-07-09-eino-06-react-agent %}))的**前提**。

还有一个贯穿全系列的事实:**core 仓库只有接口,实现全在独立的 eino-ext 仓库**。core 极小、依赖干净、可测。后面每篇标注实现位置都会区分「core 有接口」还是「实现在 eino-ext」。

> 这一节是全系列公共背景,后续 11 篇不再重复。

## 从一次事故说起:字符串拼接是怎么把系统拖垮的

Prompt 组件在教程里通常被一笔带过——「不就是字符串模板吗」。但在真实系统里,我见过的、也亲手埋过的坑,恰恰从「不就是字符串拼接」开始。

一个典型的演化路径,很多团队都走过:

**第一版**,直接 `fmt.Sprintf`:

```go
prompt := fmt.Sprintf("你是一个%s。请回答:%s", role, query)
resp := callLLM(prompt)
```

能跑。但两周后需求来了:要区分 system / user 角色,要注入多轮历史,要给不同租户配不同的系统提示。于是代码变成一坨 `strings.Builder` 拼接,role 用魔法字符串前缀区分,历史用 `\n` 手动 join。**真正的事故点在历史注入**:某次一个用户的历史消息里带了和分隔符冲突的内容,拼接出的 prompt 结构错乱,模型把用户历史当成了系统指令执行——这就是一次典型的、由「手工拼字符串」直接导致的 prompt 注入。

问题的本质是:**当「结构化的消息列表」被压扁成「一根字符串」时,结构信息就丢了**,而结构信息恰恰是安全边界。eino 的 Prompt 组件解决的就是这件事——它自始至终不让你碰那根字符串,你操作的永远是 `[]*schema.Message`。

## 结论 ✅:一等实现,源码只有一个文件但取舍很清楚

源码在 `components/prompt/chat_template.go`。**v0.8.12 只有这一个稳定的 `ChatTemplate`,没有所谓 agentic 版本的模板**——别被二手资料里「agentic prompt」之类的说法误导,core 里 grep 不到。

接口本身极简:

```go
// components/prompt 的核心抽象
type ChatTemplate interface {
	Format(ctx context.Context, vs map[string]any, opts ...Option) ([]*schema.Message, error)
}
```

一个方法,输入变量表,输出消息列表。所有花活都在实现里。构造用 `prompt.FromMessages`,配合插值方言枚举:

```go
import (
	"github.com/cloudwego/eino/components/prompt"
	"github.com/cloudwego/eino/schema"
)

tpl := prompt.FromMessages(schema.FString,
	schema.SystemMessage("你是一个{role}"),
	schema.MessagesPlaceholder("history", true), // 占位:运行期注入历史消息列表
	schema.UserMessage("{query}"),
)

msgs, err := tpl.Format(ctx, map[string]any{
	"role":    "资深 Go 工程师",
	"history": priorMessages, // []*schema.Message,而不是拼进字符串
	"query":   "解释一下 GMP 调度模型",
})
```

这里有个**关键设计**值得停下来看:`MessagesPlaceholder`。它让你把历史消息作为**结构化列表**注入,而不是拼进某个字符串变量。回到开头那次事故——如果历史走的是 `MessagesPlaceholder`,用户历史永远是独立的 `Message` 条目,带着自己的 `Role`,**没有任何机会越过角色边界去冒充 system**。这就是「消息即类型」这条主线在安全上的直接兑现。

## 三种插值方言,以及为什么方言选择是个架构决策

`FromMessages` 的第一个参数是插值格式,v0.8.12 提供三种:

| 方言 | 语法 | 适用场景 |
|---|---|---|
| `schema.FString` | `{var}` | 最简,内部自研,插值快 |
| `schema.GoTemplate` | `{% raw %}{{.var}}{% endraw %}` | 需要条件/循环/函数,复用 Go 生态 |
| `schema.Jinja2` | `{% raw %}{{ var }}{% endraw %}` | 迁移自 Python 侧、prompt 由算法同学维护 |

看起来是「支持多种模板语法」这种便利功能,但从架构视角,方言的选择其实是**信任边界的选择**:

- `FString` 只做变量替换,**不执行逻辑**。用户可控变量走这条最安全——它无法被诱导执行分支或调用函数。
- `GoTemplate` / `Jinja2` 支持逻辑,表达力强,但如果模板本身来自不可信来源,就有模板注入面。

我的生产准则很直接:**面向终端用户、变量来自用户输入的模板,一律 `FString`;需要复杂逻辑的,把逻辑上移到 compose 层用代码表达,而不是塞进模板方言**。模板负责渲染,不负责决策——这条边界一旦松掉,你的 prompt 就变成了一门没有类型检查、没有测试覆盖的第二语言。

## 和 compose 串起来:类型在编译期就对齐

Prompt 极少单独用,它是「模板 → 模型」链路的第一环。用 [compose]({% post_url 2026-07-09-eino-05-compose %}) 串起来就是最小的一个 LLM 应用:

```go
chain := compose.NewChain[map[string]any, *schema.Message]()
chain.AppendChatTemplate(tpl).AppendChatModel(model)

runnable, err := chain.Compile(ctx) // 编译期 + Compile 期双重类型检查
out, err := runnable.Invoke(ctx, map[string]any{"role": "...", "query": "..."})
```

`Compile` 会在**启动期**遍历链路,检查上游输出类型能否赋值给下游输入。模板输出 `[]*schema.Message`,模型输入正好吃这个,对得上。如果你不小心把一个输出 `string` 的节点接到模型前面,`Compile` 直接返回错误——**不是等线上第一个请求进来才 panic**。

这是 Go 静态类型给 eino 的最大红利,也是它区别于「靠 dict 传参、运行时才报 KeyError」那类 Python 框架的根本点。对一个要跑在生产、要过 CI 的系统,「错误在编译/启动期暴露」和「错误在凌晨三点的线上暴露」是两个物种。

## 一点性能视角:模板是无状态的,请复用它

`ChatTemplate` 由 `FromMessages` 构造出来后是**无状态、并发安全**的——它只持有模板定义,不持有任何请求态。`Format` 每次调用才分配新的 `[]*schema.Message`。

这意味着一条明确的实践:**模板在启动期构造一次,存全局或注入依赖,别在请求路径里反复 `FromMessages`**。虽然 `FString` 的解析成本很低,但在 QPS 上千的热路径上,「每请求重建模板」这种浪费累积起来就是实打实的 GC 压力和 CPU。`Format` 本身的开销主要是消息切片的分配,基本不可省;真正能省的是模板构造。同理,编译后的 `Runnable`([第 5 篇]({% post_url 2026-07-09-eino-05-compose %}) 会讲)也要复用,别每请求 `Compile`。

## 生产实践清单

几条从实际系统里沉淀下来的:

- **系统提示要做版本管理**:prompt 是会「回归」的资产——改一版可能悄悄拉低某类 query 的质量。把系统提示当代码管,进 git、走 review、能回滚,配合 [第 12 篇]({% post_url 2026-07-09-eino-12-evaluation %}) 的离线评测建基线。
- **插值缺变量是配置错误,不是运行时容错**:`Format` 时 map 缺了模板引用的变量会返回 error,别 recover 掉它静默渲染成空串——那会让一个配置问题伪装成「模型效果变差」,极难排查。让它在启动自检 / 冒烟测试里就炸出来。
- **历史一律走 `MessagesPlaceholder`**:不光是安全,还因为多轮历史作为结构化 `Message` 列表,才能被下游正确处理(工具消息、assistant 的 tool_calls 都要保结构)。拼进字符串等于自毁结构。
- **别在模板里塞业务分支**:条件路由、循环属于 compose 层。把 if/else 硬编进模板字符串,等于把可测试的代码逻辑降级成不可测的文本。
- **「按需加载大段指令」不是 Prompt 的活**:那是 adk 的 Skill 中间件([第 10 篇]({% post_url 2026-07-09-eino-10-skill %}))在 Prompt 之上做的渐进式披露。别把十个场景的指令全塞进一个巨型系统提示,既贵又稀释注意力。

## 小结

Prompt 是四项基础能力里最朴素的一项,但它在架构上一点都不朴素:它用 `ChatTemplate.Format → []*schema.Message` 这一个签名,确立了「组件输出即结构化消息、类型编译期对齐、结构即安全边界」的全框架基调。理解了这一条,后面 compose 的流缝合、ReAct 的封装、Skill 的注入,都是同一条主线的延伸。

| 项 | 结论 |
|---|---|
| 实现程度 | ✅ 一等,稳定 |
| 源码 | `components/prompt/chat_template.go` |
| 核心 API | `prompt.FromMessages(格式, msgs...)` + `tpl.Format(ctx, vars)` |
| 三方言 | `FString`(安全默认)/ `GoTemplate` / `Jinja2` |
| 主线价值 | 输出即 `[]*schema.Message`,结构即安全边界,类型编译期对齐 |
