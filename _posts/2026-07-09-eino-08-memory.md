> eino「逐能力核对」系列第 8 篇,第二阶段收官项 **Memory**,结论:**⚠️ v0.8.12 没有独立的 memory / session 包**。这是全系列最容易被二手资料夸大的一项,本篇专门泼冷水、划清边界。但我想更进一步:**框架没有把「记忆」做成一等模块,不是缺失,反而是一个克制且正确的架构决策**——因为记忆的本质不是「把对话存起来」,而是「在有限的上下文预算里,决定带什么、丢什么、压缩什么」,而这个策略高度业务相关,框架替你决定反而是错的。三层架构见 [第 1 篇]({% post_url 2026-07-09-eino-01-prompt %})。

## 结论:⚠️ 非一等,别当「自带记忆」的框架

我把 core 翻遍,**没有** `memory` 或 `session` 顶层模块。所谓「Memory」在 v0.8.12 只由两块拼凑:

1. **`adk/filesystem` 的 `NewInMemoryBackend()`** —— 一个纯内存的文件后端。
2. **`adk/runctx.go` / turn_buffer 的 per-run 状态** —— 单次运行(one run)内部的上下文与轮次缓冲。

就这两样。没有跨会话的持久记忆模块,没有自动的历史管理。看到「eino 支持持久化会话」这类描述,默认它指的是「你可以自己接存储」,而不是框架自带。

## 先厘清:三个被混为一谈的概念

「记忆」这个词在 Agent 语境里被用得极其含糊。生产里我坚持把三件不同的东西分开——混淆它们会导致架构错位:

| 概念 | 是什么 | 生命周期 | v0.8.12 |
|---|---|---|---|
| **per-run 状态** | 单次 Run 内多轮循环记住已发生的 | 一次 `Run`,结束即销毁 | ✅ `adk/runctx` 天然工作 |
| **跨会话记忆** | 用户历史、长期偏好,下次对话还记得 | 跨会话,需持久化 | ❌ 无一等支持,应用层自建 |
| **checkpoint** | 把未完成的 Run 存起来以便 Resume | 跨中断,直到恢复 | ✅ 但那是运行时状态,不是对话记忆([第 11 篇]({% post_url 2026-07-09-eino-11-runtime %})) |

**这三者最常被混淆的是「跨会话记忆」和「checkpoint」。** 一个存的是「用户聊过什么」(业务语义),一个存的是「一次运行卡在哪、怎么继续」(运行时语义)。把 checkpoint 当记忆用,或反过来,都会踩坑。[第 11 篇 Runtime]({% post_url 2026-07-09-eino-11-runtime %}) 会再钉一遍这个区分。

## 逐块看清边界

### ① NewInMemoryBackend:唯一的 filesystem 后端

```go
import "github.com/cloudwego/eino/adk/filesystem"

// v0.8.12 唯一的 filesystem backend 构造:纯内存
backend := filesystem.NewInMemoryBackend()
```

> ⚠️ 划重点:**v0.8.12 的 `adk/filesystem` 只 ship 了 `NewInMemoryBackend()` 这一个构造**,没有 `NewLocalBackend` / `LocalConfig`。有些资料(包括我早先的旧稿)写的磁盘后端 API 是**不存在的**。这个内存后端更像「给 DeepAgent 当草稿纸的临时文件系统」([第 9 篇]({% post_url 2026-07-09-eino-09-multi-agent %})),而不是跨会话的长期记忆——**进程重启就没了**。

### ② per-run 状态:生命周期只有一次 Run

`adk/runctx.go` / turn_buffer 维护的是**单次运行内部**的上下文与轮次缓冲。它的生命周期就是**一次 `Run`**,Run 结束即销毁。它让 ReAct([第 6 篇]({% post_url 2026-07-09-eino-06-react-agent %}))的多轮 reason-act 循环里能记住本次已经发生了什么,但**不跨 Run**。

## 核心视角:记忆是上下文预算问题,不是存储问题

现在讲本篇真正想传递的东西。很多人以为「给 Agent 加记忆」就是「把对话存进数据库,下次读出来拼回去」——好像是个存储问题。**它不是。** 存储是最简单的部分,难的是:

**上下文窗口是有限的、且越用越贵。** 模型的上下文有硬上限,而且 prefill 的成本和延迟随输入长度线性增长。如果你天真地「把全量历史每轮都拼回去」,会得到一条 O(n) 增长的曲线:

- 对话进行到第 50 轮,你每次请求都在重发前 49 轮的全文;
- token 消耗和首 token 延迟一路攀升;
- 最终撞上上下文上限,请求直接失败。

所以真正的记忆工程,核心不是「存」,是**「压」**——在固定的上下文预算内,决定:

- 哪些近期消息**原样保留**(滑动窗口);
- 哪些久远历史**压缩成摘要**(让 LLM 把前 40 轮浓缩成一段);
- 哪些关键事实**结构化抽取**存起来(用户偏好、已确认的约束),按需注入而非全量重放。

这套「带压缩的记忆策略」高度依赖你的业务:客服场景要记工单状态,陪伴场景要记用户偏好,代码助手要记项目上下文——**不存在一个通用的正确答案**。这正是我说「框架不做一等 memory 模块是对的」的理由:它把这个必须由业务决定的策略,留给了最懂业务的你,而不是塞一个通用实现让你将就。

## 生产实践:自己在应用层做,分寸自己定

跨会话记忆是**你的应用层责任**。标准骨架:

```go
// 把对话历史落 Redis/DB,下次 Run 前捞出来、压缩、拼进 AgentInput
func chatTurn(ctx context.Context, agent adk.Agent, sessionID, userMsg string) {
	history := loadHistory(ctx, sessionID)        // 从 Redis/DB 读
	history = compact(history)                    // 关键:滑动窗口 + 摘要,控预算
	msgs := append(history, schema.UserMessage(userMsg))

	iter := agent.Run(ctx, &adk.AgentInput{Messages: msgs})
	// ... 消费事件流,收集 assistant 回复 ...

	saveHistory(ctx, sessionID, append(msgs, assistantReply)) // 写回
}
```

那行 `compact()` 才是记忆工程的重心——`load` / `save` 是体力活,`compact` 是设计活。几条实践:

- **别无脑全量重放**:全量历史会越滚越长撑爆上下文、抬高成本。生产里默认做滑动窗口 + 摘要。
- **摘要也要版本管理**:摘要是 LLM 生成的、有损的。摘要 prompt 改一版,记忆行为就变。把它当 [第 12 篇]({% post_url 2026-07-09-eino-12-evaluation %}) 的评测对象之一。
- **结构化事实单独存**:用户的硬约束(「我对花生过敏」)不该指望摘要保住,应结构化抽取、稳定注入,别赌它没被压缩掉。
- **和 checkpoint 严格分开**:再强调一次——[第 11 篇]({% post_url 2026-07-09-eino-11-runtime %}) 的 checkpoint 是运行时状态持久化,不是对话记忆。两套存储、两种语义,别合并。
- **未来方向**:更完整的 memory/session 抽象是后续版本方向,不在 v0.8.12。选型按 v0.8.12 的真实边界评估。

## 小结

eino v0.8.12 给你的是「一次 Run 内的 per-run 状态 + 一个内存文件后端」,真正的持久化记忆要靠 `AgentInput.Messages` 的手动装填 + 你自己的存储层。但别把这当缺陷来抱怨——记忆的核心难点是**上下文预算内的压缩策略**,而这个策略必须由业务定义。框架把「存」留给你、把「压」的自由留给你,是一个诚实且克制的边界。把记忆当预算问题设计,而不是当存储问题外包。

| 项 | 结论 |
|---|---|
| 实现程度 | ⚠️ 非一等 |
| 仅有 | `adk/filesystem.NewInMemoryBackend()` + `adk/runctx` per-run 状态 |
| 没有 | 独立 memory/session 包、磁盘后端、跨会话持久记忆 |
| 核心难点 | 不是存储,是上下文预算内的压缩(滑动窗口 + 摘要 + 结构化事实) |
| 持久化 | 应用层自建(Redis/DB + 手动装填 `AgentInput.Messages`) |
